# 背景 — manabi-repeat の収束と、そこから得た次のテーマ

## manabi-repeat とは何だったか

[yktsnd/manabi-repeat](https://github.com/yktsnd/manabi-repeat) は、写真で撮った小学生向け算数問題から解法構造を抽出し、同じ考え方で解ける練習問題を生成し、機械的に検証する PoC だった。

このテーマは最初から技術的に難しいことを前提に進めた。写真入力、問題文の読み取り、解法構造の抽出、同型問題の生成、生成後の検算、品質ゲート、プライバシー配慮、β公開準備まで含めて、**難しさそのものを検証対象**にしていた。

## 収束の判断(2026-07)

技術的困難は一定程度アプローチ可能だった。しかし、その困難を乗り越えるための**開発・検証・運用・サポート負荷**に対して、教育プロダクトとしての市場価値・継続利用価値・勝ち筋が十分に見えなかった。よって PoC 段階で区切り、**manabi-repeat 本体は商用教育 AI プロダクトとしての継続開発を終了**する。

## PoC を通じて確信したこと

manabi-repeat で本当に重かったのは「AI に問題を解かせる・作らせる」部分ではなく、**プロトタイプを「公開できるアプリ」に引き上げる過程**だった。

いま、AI API や生成 AI を使って試作品を作れる人は急速に増えている。一方で、それを実際にアプリとして公開する段階には、共通してこういう障壁がある:

1. **API キーの壁** — キーはクライアントに置けないので、試作品がどれだけ動いていても公開には必ずバックエンドが要る
2. **課金暴走の壁** — 従量課金 API は、悪意・バグ・想定外のバズのどれでも財布に直結する。上限を「確実に」効かせる仕組みを自作する必要がある
3. **悪用の壁** — 認証のない LLM エンドポイントは即座に「無料プロキシ」として発見・濫用される
4. **定型コードの壁** — ストリーミング、リトライ、エラーマッピング、CORS。毎回同じものを書く
5. **プライバシーの壁** — ユーザー入力をそのままログに残すと、それだけで負債になる
6. **運用の壁** — 誰がいくら使ったか、いつ止めるか、を把握・制御する手段

この「**AI を使える**」と「**AI アプリとして出せる**」の間にある障壁を、小さく実用的な道具で解決する方向が、多くの人に必要とされるはずだ — というのが manabi-repeat から持ち帰る結論であり、次のプロジェクト **sekimori** の出発点になる。
