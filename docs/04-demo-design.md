# デモ設計 — 「守りが効く瞬間」を見せる

設計: Fable 5(2026-07)。実装: Sonnet 5 に委託。DX レビュー([03-dx-review.md](03-dx-review.md))の A-4・A-6・B-3 に対応する。

## 0. デモが証明すべきこと

sekimori のデモは「チャットが動くこと」を見せるものではない。チャットが動くだけなら sekimori は不要である。見せるべきは**遮断が正しく起きる瞬間**、すなわち:

1. トークンなしの侵入が 401 で止まる(野良プロキシ化しない)
2. 予算を使い切った利用者が 429 で止まり、**いつ再開できるかが伝わる**
3. 連打が 429 で止まる(レート制限)
4. 許可外モデルが 403 で止まる(勝手に高いモデルを使われない)
5. 失効させたトークンが即座に無効になる(招待の取り消しが効く)
6. その間、正常な利用者は普通にチャットできている

この 6 点が 1 コマンドで、実 API キーなし・課金ゼロで再現できること。

## 1. `examples/demo.sh` — ワンコマンド・シナリオデモ

### 実行形態

```bash
cd projects/sekimori && npm install   # 初回のみ
bash examples/demo.sh
```

- 前提: bash / curl / node(npm install 済み)。それ以外の依存なし。jq は使わない
- 擬似上流(`examples/mock-upstream.mjs`)と sekimori 本体を一時ディレクトリの config で起動し、終了時に trap で必ず片付ける。作業ファイルは `mktemp -d` 配下に置き、リポジトリを汚さない
- ポートは既定(上流 19999 / sekimori 18787 など通常使われない番号)とし、使用中なら失敗してよい(メッセージで案内)

### シナリオ(3 幕構成)

出力は幕ごとに見出しを付け、各ステップで「何を証明したか」を 1 行添える。

**第 1 幕「公開する」**
1. 擬似上流を起動(ここが本物の Anthropic API の代役であることを説明)
2. sekimori を起動。config は demo 用に生成: 月次上限 $5、レート制限 5 req/min、モデル 1 つ(`claude-haiku-4-5-20251001`、参考価格)
3. 起動サマリ(DX レビュー A-3 で追加するもの)がそのまま画面に出る — 「何を守るか」を宣言する演出を兼ねる

**第 2 幕「招待する」**
4. `alice` のトークンを発行(dailyUsd: 1.0 — 普通の利用者)
5. `mallory` のトークンを発行(dailyUsd: 0.001 — すぐ上限に達する設定であることを説明)
6. alice が非ストリームで 1 往復 → 応答が返る
7. alice がストリーミングで 1 往復 → テキストが流れる
8. alice の `/v1/usage` → 消費額が記録されている

**第 3 幕「守りが効く」**
9. トークンなしで `/v1/messages` → **401**
10. alice が許可外モデルを指定 → **403**
11. mallory が 1 回使う → 成功。もう 1 回 → **429 budget_exceeded_error + Retry-After**(「明日の UTC 0 時まで」であることを表示)
12. alice が 6 連打 → 6 回目が **429 rate_limit_error + Retry-After**
13. 管理者が `/admin/usage` で全体を確認 → mallory の消費が見える
14. 管理者が mallory を失効 → mallory の次のリクエストが **401**
15. まとめ表示: 「この間、上流に漏れたのは alice と mallory の正当なリクエストだけ。API キーは一度もクライアントに渡っていない」

### スモークテストとしての性質(B-3)

- 各ステップは期待 HTTP ステータスを検証し、**1 つでも不一致なら非ゼロ終了**する
- つまり demo.sh は「見せる台本」と「E2E スモークテスト」を兼ねる。CI 導入時にはそのまま 1 ジョブになる

### 実 API モード(おまけ)

`SEKIMORI_DEMO_REAL=1 ANTHROPIC_API_KEY=sk-... bash examples/demo.sh` で擬似上流の代わりに本物の Anthropic API を使う。シナリオは同一(消費は数セント以下に収まる設計: max_tokens を小さく)。既定は必ずオフライン。

## 2. `examples/chat.html` — リファレンスクライアント化

現状の chat.html は「URL・トークン・モデルの 3 項目をエンドユーザーに入力させる」形で、統合パターンのお手本として誤っている(DX レビュー A-4)。次の形に改める。

### 設計原則

**このファイルは「コピーして自分のアプリの出発点にするもの」**。だから、開発者が触る場所とエンドユーザーが触る場所をコード上で分離して見せる。

- ファイル冒頭の `<script>` に開発者向け CONFIG ブロックを置く:
  ```js
  // ここだけ書き換えれば自分のアプリになる(エンドユーザーはこの下を触らない)
  const CONFIG = {
    baseUrl: "http://localhost:8787",
    model: "claude-haiku-4-5-20251001",
    appName: "My AI App",
    maxTokens: 512,
  };
  ```
- エンドユーザーの入力は**招待トークンのみ**。初回に入力させ、`localStorage` に保存。以後は入力欄を出さない(「トークンを変更」リンクで再入力可能)
- 会話はページ内で複数ターン保持(配列を `messages` としてそのまま送る。リロードで消えてよい)

### 上限体験の設計(A-6)

- 送信のたびに `/v1/usage` を取得し、「今日の利用: $0.012 / $1.00」を小さく常時表示する
- エラーは `error.type` で分岐した**利用者の言葉**で表示する:
  - `authentication_error` → 「招待トークンが無効です。配布元に確認してください」+トークン再入力欄
  - `budget_exceeded_error` → 「今日の利用上限に達しました。あと約◯時間で再開します」(`Retry-After` から計算)
  - `rate_limit_error` → 「送信が速すぎます。◯秒待ってください」(`Retry-After`)
  - その他/通信エラー → 「接続できません。運営者に連絡してください」
- 生の HTTP ステータスや JSON は既定で見せない(details で折りたたみ表示は可)

### 変えないこと

依存ゼロ・1 ファイル・ビルドなし。フレームワーク導入は不可(「出発点としてコピーできる」性質が失われる)。
