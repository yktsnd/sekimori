# sekimori(関守)— コンセプト

> **凍結済みの歴史文書**(ラウンド記録)。コンセプト自体は現在も有効だが、§8 のロードマップは [ROADMAP.md](../../ROADMAP.md) に引き継がれ、そちらが一次情報源。

> **AI プロトタイプを、API キーを晒さず・予算を溶かさず・悪用されずに公開するための、最小のセルフホスト型ゲートウェイ。**

名前は江戸時代の「関所の番人(関守)」から。ユーザーと LLM API の間に立ち、通行手形(トークン)を検め、通行量(予算)を管理する。

## 1. 解く課題

ローカルでは `.env` にキーを置けば AI プロトタイプは動く。それを他人に使わせようとした瞬間、次の全部が同時に必要になる:

| 障壁 | 放置した場合に起きること |
|---|---|
| キー秘匿 | クライアント配布 = キー流出。即座に濫用される |
| 予算上限 | バグ・悪意・バズのいずれでも青天井の請求 |
| 利用者認証 | 認証なしエンドポイントは「無料 LLM プロキシ」として自動発見・濫用される |
| モデル/プロンプト制限 | 自分のアプリのつもりが、任意用途の汎用 API を無償提供していることになる |
| ログとプライバシー | ユーザー入力の生ログがそのまま負債になる |
| 定型コード | ストリーミング・エラー処理・CORS を毎回自作 |

どれも個別には難しくないが、**全部そろえないと公開できない**。そして全部そろえる作業は、プロトタイプ本体より重いことが多い。

sekimori はこの「公開前の必須セット」を、**設定ファイル 1 枚 + プロセス 1 個**で提供する。

## 2. 想定ユーザー

- AI API で動くものは作れた個人開発者。「週末に作ったこれを、友人・SNS・β ユーザー 10〜1,000 人に触らせたい」段階の人
- 明示的に**対象外**: チーム/企業の本番運用、マルチテナント SaaS、高トラフィック

この「個人が身内に公開する」規模に絞ることが設計のすべてを単純にする:DB 不要、水平スケール不要、ダッシュボード不要。

## 3. 提供価値(コア機能)

1. **キー秘匿** — 上流 API キーはサーバーにだけ置く。クライアントには sekimori が発行する招待トークンだけを渡す
2. **確実に効く予算上限** — 全体の月次上限(キルスイッチ)と、トークンごとの日次上限。会計が書き込めない状態では**遮断側に倒す(fail-closed)**
3. **招待トークン認証** — `curl` 一発でトークンを発行し、URL やメッセージで配る。取り消しも一発
4. **濫用防止** — モデル許可リスト、レート制限、system プロンプトのサーバー側固定(pin)
5. **プライバシー既定値** — 既定ではリクエスト/レスポンス本文をログに残さない。残すのは件数・トークン数・コストだけ
6. **ストリーミング素通し** — SSE をそのまま中継しつつ、usage だけ抜き取って会計する

## 4. 既存ソリューションとの違い

同じ領域には強力なツールが既にある。sekimori の差分は「**エンドユーザー向けの認証と割当を、個人規模向けに、デプロイ 1 単位で**」に尽きる。

| 既存 | 得意なこと | 個人の公開用途に足りないこと |
|---|---|---|
| LiteLLM Proxy | 100+ プロバイダ統合、virtual key、予算管理 | チーム/企業向け。予算管理に Postgres が要り、設定項目が膨大。「週末アプリ」には過剰 |
| Cloudflare AI Gateway | 観測・キャッシュ・レート制限 | エンドユーザー認証と個人別割当は守備範囲外。結局自前 BFF が要る |
| Helicone / Portkey | 観測性、プロンプト管理(SaaS) | 同上。またログが外部 SaaS に渡る |
| OpenRouter | API 統合と BYOK | 自アプリの利用者管理・割当は自前 |
| Vercel AI SDK | クライアント/サーバーの実装 DX | 認証・予算・濫用対策は「自分で書く」側に残る |
| Firebase AI Logic | クライアントから Gemini を安全に呼ぶ managed 経路 | Google/Gemini エコシステム限定。この製品の存在自体が「同じ課題をプラットフォーマーも見ている」傍証 |

つまり既存ツールは「API を束ねる」「観測する」が主眼で、**「自分のアプリの利用者を検める」部分が空いている**。そこが sekimori の場所。

## 5. 非目標(やらないこと)

- マルチテナント SaaS 化、課金代行(Stripe 連携)
- ダッシュボード UI(MVP は curl + JSON で足りる)
- プロンプト管理・評価(evals)・キャッシュ・ルーティング最適化
- 100+ プロバイダ対応(MVP は Anthropic Messages API のみ。OpenAI 互換は次段)
- 水平スケール(単一プロセス前提を明記する)

## 6. 検討して見送った代替コンセプト

- **公開準備オーディター CLI**(リポジトリを走査して「キー露出・上限なし・PII ログ」等を検出し雛形を生成する "AI アプリの Lighthouse")— 価値はあるが、検出の網羅性が売りになる種類の道具で、機械的に正しさを検証しにくい。ゲートウェイが立ち上がった後のロードマップ候補として残す
- **コスト見積もり/予算ライブラリ単体** — 課題が狭すぎ、ゲートウェイの一機能に吸収される

## 7. リスクと、それをどう検証するか

| リスク | 検証方法 |
|---|---|
| 「プロキシは既に飽和」— 差分が伝わらない | README の一行目で「誰の・どの瞬間の道具か」を言い切る。LiteLLM で足りる人には LiteLLM を薦める文言を README に書く |
| 価格表の保守負荷(モデル追加・改定) | 価格はユーザー設定に置き、同梱の既定値は「参考値」と明記。未知モデルは fail-closed |
| 上流 API の仕様追従 | MVP は素通し(passthrough)設計にし、リクエスト変換を最小化する |
| 自分すら使わない道具になる | **ドッグフーディングを成功条件にする**: 自分のプロトタイプ 1 本を sekimori 経由で実際に公開し、「clone からデプロイまで 30 分以内」を計測する |

## 8. ロードマップ(MVP 後)

順序は DX レビュー([03-dx-review.md](03-dx-review.md))の結論を反映している。「最初の 5 分」を縮めるものを、デプロイ先の選択肢を増やすものより優先する。

1. npm 配布(`npx sekimori` 一発起動)+ `sekimori init` 対話ウィザード(config 生成)
2. デプロイガイド(Fly.io / Railway / VPS+Caddy。ドッグフーディングで実測しながら書く)
3. OpenAI 互換上流(`/v1/chat/completions`)— Store と並ぶ公式拡張点として抽象化とセットで
4. Cloudflare Workers + Durable Objects デプロイ(エッジで動く 1 コマンド公開)
5. BYOK モード(利用者が自分のキーを持ち込む。開発者の予算リスクをゼロにする選択肢)
6. マジックリンク/OAuth による利用者認証(招待トークンの上位互換)
7. 簡易利用状況ページ(静的 HTML 1 枚)
8. `npm create sekimori` スキャフォールド(フロント雛形込み)
9. 公開準備オーディター(見送り案の再回収)
