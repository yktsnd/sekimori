# DX レビュー — 利用者・コントリビューターのハードルを設計思想レベルで点検する

> **凍結済みの歴史文書**(ラウンド記録)。「今すぐ直す」7 件は実装済み。「公開前必須」項目は [05-sustainability-review.md](05-sustainability-review.md) の分析を経て [ROADMAP.md](../../ROADMAP.md) に引き継がれた。

実施: 2026-07(設計・レビュー: Fable 5)。対象は MVP 実装時点の sekimori 全体。
「利用者(sekimori を導入して自分のアプリを公開する開発者)」と「コントリビューター(OSS として参加する第三者開発者)」の 2 視点で、実際の利用フローを一歩ずつ歩いて摩擦を洗い出した。

判定の凡例: **[今すぐ直す]** = 本ブランチで実装委託 / **[公開前必須]** = リポジトリ独立・公開時のゲート条件 / **[ロードマップ]** = 優先順位の変更として記録

---

## A. 利用者(導入する開発者)の旅

### A-1. 発見 → 最初の 5 分

- **現状**: リポジトリを clone し、`npm install` して `npx tsx src/main.ts` で起動する。パッケージ配布がなく、「試す」の入口が「開発環境を作る」になっている。
- **問題(思想レベル)**: sekimori の想定ユーザーは「プロトタイプは作れたが公開段階で詰まっている人」。その人に最初に要求する行為が git clone なのは、プロダクトの主張(公開の障壁を下げる)と自己矛盾する。
- **判定**: **[公開前必須]** npm パッケージ化(`dist/` ビルド + `npx sekimori` bin)。`npx sekimori@latest` 一発で起動できることを公開時の受け入れ条件にする。**[ロードマップ]** `sekimori init` 対話ウィザード(config 生成)を Workers 対応より上位に繰り上げる。理由: エッジ対応は「デプロイ先の選択肢」を増やすが、init は「最初の 5 分」を縮める。最初の 5 分の方が離脱に直結する。

### A-2. config の初期作成

- **現状**: example をコピーして JSON を手で編集。モデル名と価格(USD/MTok)を自分で知っている必要がある。
- **問題**: 価格を調べる作業は本質でない摩擦。また config を用意し忘れて起動すると、ファイル不在のエラーがそっけない。
- **判定**: **[今すぐ直す]** config ファイル不在時のエラーメッセージを「example のコピー方法と README の該当節」を示す案内文にする。example config には主要モデルの参考価格を現行値で入れておき「参考値であり要確認」と明記(既に方針あり、維持)。価格の自動取得はしない(fail-closed 原則: 価格の真実はユーザーの確認に置く)。

### A-3. 起動した「あと」の不安

- **現状**: 起動ログは listen アドレスと upstream の 1 行のみ。自分の設定(予算・モデル・CORS)が効いているのかは、失敗して初めて分かる。
- **問題**: sekimori の価値は「守りが効いていること」なのに、起動時点でそれが見えない。守りの道具は「何を守っているか」を自分から言うべき。
- **判定**: **[今すぐ直す]** 起動時に有効設定のサマリ(ポート、上流、許可モデル、月次/日次予算、レート制限、CORS 許可 Origin、ストア種別、logBodies)を数行で表示する。秘密情報は出さない。

### A-4. 友人を招待する瞬間

- **現状**: curl でトークンを発行し、トークン文字列を渡す。受け取った側は chat.html で「ベース URL・トークン・モデル名」の 3 項目を入力する。
- **問題**: エンドユーザー(友人)に URL とモデル名まで入力させるのは統合パターンとして誤ったお手本。実際のアプリでは URL とモデルは開発者が埋め込み、ユーザーが持つのは招待トークンだけのはず。example が「正しい使い方」を教えていない。
- **判定**: **[今すぐ直す]** chat.html を「リファレンスクライアント」に格上げする(詳細は [04-demo-design.md](04-demo-design.md))。開発者が編集する CONFIG ブロック(URL・モデル)と、エンドユーザーが一度だけ入れる招待トークン(localStorage 保持)を分離する。

### A-5. CORS でつまずく瞬間

- **現状**: ブラウザから叩くには `cors.allowedOrigins` の編集と再起動が必要。忘れた場合、ブラウザ側には CORS の一般的エラーしか出ず、サーバー側 stdout には何も出ない。
- **問題**: 「どちらの画面を見ても原因が書かれていない」状態が起きる。fail-closed の思想は正しいが、**遮断したことを運営者に教えない遮断**は、思想ではなくただの不親切。
- **判定**: **[今すぐ直す]** 許可されていない Origin からのリクエストを受けた際、サーバー stdout に 1 行の警告(`blocked origin: ... allowedOrigins に追加が必要かもしれない`)を出す。遮断自体は変えない。

### A-6. 上限に当たった利用者の体験

- **現状**: 予算超過は 429 + `budget_exceeded_error`。「いつ再開できるか」は返らない。
- **問題**: 守りが効いた瞬間こそ、この道具の価値が試される瞬間。エンドユーザーに「壊れた」と思わせたら、招待した開発者の信用が傷つく。日次上限は UTC 深夜に必ず解除されるのだから、機械可読に伝えられる。
- **判定**: **[今すぐ直す]** 予算超過 429 に `Retry-After` を付ける(日次 → 次の UTC 0 時までの秒数、月次 → 翌月 1 日 UTC 0 時までの秒数)。リファレンスクライアントはこれを「今日の上限に達しました。あと◯時間で再開します」と表示する。

### A-7. 本番公開(インターネットに出す)の道筋

- **現状**: HTTPS・デプロイ先の話が README にない。localhost で完結している。
- **問題**: 「公開の障壁を下げる」道具なのに、最後の一歩(公開)がドキュメント上に存在しない。
- **判定**: **[公開前必須]** デプロイガイド(`docs/deploy.md`: Fly.io / Railway / VPS+Caddy の 2〜3 レシピ、HTTPS 前提の明記、`SEKIMORI_ADMIN_KEY` の生成方法)。本ブランチでは書かない(実環境で検証せずに書くデプロイ手順は負債になるため。ドッグフーディング時に実測しながら書く)。

## B. コントリビューター(第三者開発者)の旅

### B-1. 言語の壁

- **現状**: README・docs・コードコメントがすべて日本語。
- **問題**: OSS としての最大の参加障壁。日本語話者以外には issue を立てることすら難しい。
- **判定**: **[公開前必須]** リポジトリ独立時に英語を一次言語にする(README は英語、`README.ja.md` を併設。コードコメントは英語化)。playground 内にある間は日本語のままでよい(現段階の読者は自分たちだけ)。

### B-2. 「実装契約」という統治の見え方

- **現状**: `02-mvp-spec.md` は「実装者はスコープを勝手に広げない」という内部委託用の契約文書。これを外部の人が読むと「機能提案は歓迎されない閉じたプロジェクト」に見える。
- **問題**: 非目標(やらないこと)を明確にするのは参加障壁を**下げる**優れた仕掛けだが、それは「あなたの PR が無駄になる前に教える」という書き方をされたときだけ。契約の言葉のままだと逆に働く。
- **判定**: **[公開前必須]** CONTRIBUTING.md を用意し、(1) 歓迎される PR(バグ修正・テスト・ドキュメント・Store 実装の追加)、(2) 先に issue を立ててほしいもの(機能追加)、(3) 非目標(お互いの時間を守るための一覧)、(4) レビュー基準は「迷ったら fail-closed」の 4 点で構成する。`02-mvp-spec.md` は「MVP 時点の実装契約(歴史文書)」と冒頭に明記して凍結する。

### B-3. 貢献の検証手段

- **現状**: `npm test`(オフライン・実キー不要・33 件)と `tsc --noEmit`。CI はない。
- **評価**: テストが実キーなしで回るのは参加障壁を大きく下げる設計であり、この思想は守るべき資産。ただし CI がないと、コントリビューターは「自分の環境では通った」以上の確信を持てず、メンテナは全 PR を手元で検証する羽目になる。
- **判定**: **[公開前必須]** リポジトリ独立時に GitHub Actions(test + typecheck のみ、数十秒で終わる最小構成)。**[今すぐ直す]** `examples/demo.sh`(後述)を「期待ステータスの不一致で非ゼロ終了する」形にし、スモークテストを兼ねさせる。

### B-4. 拡張点の不在(見えない)

- **現状**: `Store` インターフェースは事実上の拡張点(Redis / SQLite / KV)だが、どこにも「ここが拡張点だ」と書かれていない。
- **問題**: コントリビューターが最初に探すのは「どこなら手を入れて歓迎されるか」。それが明示されていないと、コアの設計原則(fail-closed、依存最小)に踏み込む PR が来て、お互いに消耗する。
- **判定**: **[公開前必須]** CONTRIBUTING.md に「サポートされる拡張点は Store と上流プロバイダ」と明記。**[ロードマップ]** 上流プロバイダの抽象化(OpenAI 互換対応)は、この「拡張点の公式化」とセットで行う。

### B-5. 依存ゼロ・単一プロセスという思想の防衛

- **評価(問題なし)**: 「ランタイム依存は hono のみ」「単一プロセス前提」は、コントリビューターにとって理解コストが低く、レビューが速く、壊しにくい。これは参加障壁を下げる側に働いている。**この制約自体への変更提案(DB 追加・スケール対応)は非目標であることを CONTRIBUTING に書くことで防衛する。**

## C. 決定まとめ

### 今すぐ直す(本ブランチで Sonnet 5 に実装委託)

1. 起動時の有効設定サマリ表示(A-3)
2. config 不在時の案内つきエラー(A-2)
3. 予算超過 429 への `Retry-After` 付与(A-6)+テスト
4. 非許可 Origin 受信時のサーバー側警告ログ(A-5)
5. chat.html のリファレンスクライアント化(A-4、設計は [04-demo-design.md](04-demo-design.md))
6. `examples/demo.sh` ワンコマンドデモ = スモークテスト(B-3、同上)
7. README 更新(デモ節の追加、Retry-After の記載)

### 公開前必須(リポジトリ独立時のゲート条件)

npm パッケージ化(`npx sekimori`)/ 英語一次言語化 / LICENSE / CONTRIBUTING.md(B-2・B-4 の構成)/ 最小 CI / デプロイガイド / 名称の正式チェック

[01-concept.md](01-concept.md) §8 を改訂: `sekimori init` ウィザードと npm 配布を最上位へ。Workers 対応はその後。
