# サステナビリティ・レビュー — HYPE ではなく「末永く使われる」ための逆算分析

実施: 2026-07(設計・レビュー: Fable 5)。対象はリポジトリ独立直後(MVP + DX レビュー「今すぐ直す」対応済み)の sekimori 全体。

DX レビュー([03-dx-review.md](03-dx-review.md))が「利用フローの摩擦」を点検したのに対し、本レビューは**プロジェクトの寿命**を点検する。問いは一つ: 「3 年後も、誰かが安心して `npx sekimori` を叩ける状態を、何が壊しうるか」。

---

## 1. 理想の姿(あるべき姿)

sekimori の性質 — 個人メンテナ・ニッチな防御ツール・依存最小 — を前提にすると、「末永く使われる」の定義はスター数や機能数ではない。次の 5 条件である。

1. **信頼できる**: セキュリティ境界に置く道具なので、「守れていること」が第三者に継続的に証明されている。テストは実キーなしで誰でも回せ、CI が常時グリーンを示し、脆弱性の報告窓口がある。
2. **5 分で試せ、30 分で公開できる**: 想定ユーザー(公開段階で詰まっている個人開発者)が、メンテナに接触せずに試用から本番公開まで到達できる。
3. **メンテナが不在でも死なない**: 依存が最小(hono のみ)で腐りにくく、上流 API 追従は passthrough 設計で最小化され、ドキュメントが自己完結し、第三者が引き継げる程度に貢献経路が整っている。
4. **完成に近づく(done software)**: 機能を足し続けないと死ぬプロジェクトではなく、スコープが固定され「完成」に収束する。非目標が明文化され、スコープ拡大圧力から設計思想(fail-closed・単一プロセス・依存最小)が防衛されている。
5. **誠実に発見される**: README の 1 行目で「誰の・どの瞬間の道具か」を言い切り、過剰な人には LiteLLM を薦める。HYPE ではなく適合で選ばれる。

このうち 4 と 5 は既に設計に組み込まれている(非目標の明文化、LiteLLM への誘導)。問題は 1・2・3 である。

## 2. 問題点と真因

### 観測される問題点

| # | 問題 | 理想条件との乖離 |
|---|---|---|
| P1 | 配布形態がない(git clone + tsx 起動のみ) | 条件 2。「公開の障壁を下げる」という主張との自己矛盾(DX レビュー A-1 指摘済み・未対応) |
| P2 | 全ドキュメント・全コメントが日本語 | 条件 3。日本語話者以外は issue を立てることすら困難(B-1 指摘済み・未対応) |
| P3 | CI がない | 条件 1。グリーンの証明がメンテナの手元にしか存在しない(B-3 指摘済み・未対応) |
| P4 | ガバナンス文書がない(CONTRIBUTING / SECURITY / CHANGELOG / issue テンプレート) | 条件 1・3。貢献経路と脆弱性報告窓口が存在しない |
| P5 | ドキュメントが時系列構造(00→04 の経緯記録)で、「現在の仕様」の一次情報源が README 1 枚に肥大化している | 条件 3。README がランディング・設定リファレンス・API リファレンス・実装判断メモを兼ね、読者のタスク(試す/導入する/貢献する)と構造が一致しない |
| P6 | 公開の最後の一歩(デプロイ手順・HTTPS)がドキュメントにない | 条件 2(A-7 指摘済み・意図的未対応) |
| P7 | ドッグフーディング未実施 | 条件 5。コンセプト([01-concept.md](01-concept.md) §7)で「成功条件」と定義した「30 分で公開」が未計測の主張のまま |
| P8 | 時間経過で腐る情報(モデル名・価格)が example config と README に静的に埋まっている | 条件 3。放置されたリポジトリに見える最初の兆候になる |

### 真因の特定

P1〜P8 は独立の不具合ではない。束ねると真因は 4 つに収束する。

- **R1: プロジェクトが「作る」フェーズの読者(自分たち)向けの成果物のまま、「配る」フェーズに入っていない。** P1(配布なし)・P2(日本語)・P4(ガバナンスなし)・P6(デプロイ導線なし)はすべて「読者 = 設計者と実装委託先」という前提の残骸である。リポジトリは独立したが、成果物の宛先がまだ切り替わっていない。
- **R2: 信頼の根拠が属人化している。** テスト設計(実キー不要・オフライン完結)は資産だが、その実行がメンテナのローカルに閉じている(P3)。守りの道具は「動く」ではなく「守れていることが第三者に見える」必要がある。
- **R3: ドキュメントの構造原理が「開発の経緯」であって「読者のタスク」ではない。** P5 はこの帰結。経緯の記録(なぜこう作ったか)は消すべきでない資産だが、それが現在形の仕様書の座を占めている限り、README が肥大し続け、仕様の一次情報源が「README とソースコード」という曖昧な二重体制のままになる。
- **R4: 価値仮説が未検証のまま整備が先行するリスク。** P7。「30 分で公開できる」が実測されていない以上、配布整備(R1 対応)が済んでも「実は公開できない道具」である可能性が残る。ここだけは課金・デプロイ操作を伴うため人間(yktsnd)にしか検証できない。
- (P8 は R1 の派生: 「参考値・要確認」の明記という既存方針を、仕様として文書に固定すれば足りる)

### なぜ HYPE 型の失敗ではなく、この型の失敗を警戒するのか

sekimori は機能的には既に誠実に絞られており、「話題先行で中身が薄い」型の失敗リスクは低い。このプロジェクトの現実的な死因は逆で、**「良い道具だが、作者以外に配られる形になる前に更新が止まり、日本語の README と clone 手順だけが残る」**という静かな失敗である。本レビューの対策はすべてこの死因に向ける。

## 3. 解決方針(真因ごと)

| 真因 | 方針 | 内容 |
|---|---|---|
| R1 | **「配る」フェーズへの一括移行** | npm パッケージ化(`npx sekimori`)、英語一次言語化(README 英語 + `README.ja.md` 併設、コード内も英語)、CONTRIBUTING / SECURITY / CHANGELOG / issue・PR テンプレート整備 |
| R2 | **信頼の証明の自動化** | 最小 CI(test + typecheck + demo スモーク + pack 起動スモーク)。数十秒で終わる構成を守り、CI 自体が保守負債にならないようにする |
| R3 | **ドキュメントのゼロベース再構成** | 構造原理を「読者のタスク」に変える。README = ランディング(誰向けか・5 分で試す・次はどこへ)。`docs/configuration.md`・`docs/api.md`・`docs/design.md` = 現在形の一次情報源。`docs/history/` = 経緯の記録(凍結、日本語のまま保持) |
| R4 | **人間ゲートの明示管理** | ドッグフーディング(実デプロイ + 30 分計測)・デプロイガイド執筆(実測とセット。未検証手順は書かない原則を維持)・npm publish・商標確認を、`needs-human` ラベルの issue として管理し、「エージェントが完了させられる作業」と混ぜない |

方針の優先順位は DX レビューの結論を踏襲する: 「最初の 5 分」を縮めるもの(npm 配布・init)> 信頼の証明(CI)> デプロイ先の拡充。

## 4. 目標設定

**ゴール G: 「第三者が、メンテナに一度も接触せずに、(a) 5 分で試し、(b) 30 分で公開し、(c) 迷わず貢献できる」状態。**

これを 2 段のマイルストーンに割る。詳細と受け入れ条件は [ROADMAP.md](../../ROADMAP.md) を一次情報源とする。

- **v0.2 "distribution-ready"(エージェント作業で完結)**: (a) と (c) の成立。`npx sekimori`(pack tarball からの起動検証まで)、`sekimori init` ウィザード、英語一次言語化、docs 再構成、ガバナンス文書、最小 CI。
- **v0.3 "public release"(人間ゲート)**: (b) の成立と公開そのもの。ドッグフーディング実測、デプロイガイド、npm publish、名称・商標確認。

実行体制は従来どおり: 設計・レビュー・監督 = Fable 5、実装 = Sonnet 5 への委託、公開・課金・命名の最終決定 = 人間(yktsnd)。進行は issue 駆動とし、各 issue に受け入れ条件を明記する。
