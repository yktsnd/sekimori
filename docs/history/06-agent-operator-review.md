# エージェント・オペレーター・レビュー — 「使うのはエージェント、守られるのは人間」への転回

実施: 2026-07(設計・レビュー: Fable 5)。対象は v0.2 "distribution-ready" 完了時点の sekimori 全体。

第 1 ラウンドのサステナビリティ・レビュー([05-sustainability-review.md](05-sustainability-review.md))は「第三者の開発者がメンテナに接触せず使える」ことを理想像に置いた。本レビューはその前提を一段深く修正する、オーナー(yktsnd)からの新しい前提認識に基づく:

> **個人開発者のほとんどは、自分も含め、デプロイ方法や課金の仕組みを知らない。今後のアプリ開発は「〜というアプリを作って」とエージェントに投げ、あとはエージェント任せになる。**

## 1. 前提の転回が意味するもの

第 1 ラウンドまでの sekimori は「README を読み、curl を叩き、config を編集する人間の開発者」を利用者として想定していた。新しい前提では、この人物はほぼ存在しない。実際の登場人物は 3 者に分かれる:

| 役割 | 実体 | sekimori に触れる形 |
|---|---|---|
| **オーナー** | 人間。アプリのアイデアと財布(API キー・課金アカウント)を持つが、デプロイ・課金の仕組みは知らない | 触れない。「いくらまでなら溶けていいか」を答え、結果の要約を読むだけ |
| **オペレーター** | コーディングエージェント(Claude Code 等)。アプリを書き、sekimori を設置し、デプロイし、運用する | インストール・config 生成・起動・トークン発行・監視のすべて |
| **エンドユーザー** | オーナーの友人・SNS・β ユーザー | 招待トークンでアプリを使うだけ(従来どおり) |

この転回はコンセプトを壊さない。むしろ**強化する**。オーナーがエージェントの成果物を監査できない世界では、「アプリのコードが何をしていようと、上流キーは渡らず、支出はここで止まる」という**アプリの外側にある独立した安全境界**の価値はいっそう上がる。sekimori が守る相手は、悪意あるエンドユーザーだけでなく、**オペレーター(エージェント)の誤設定・バグからオーナーの財布を守る**役割も担う。fail-closed 設計はこの用途にそのまま適合する。

## 2. 理想の姿(改訂)

第 1 ラウンドの 5 条件(信頼・即時性・不死性・完成収束・誠実な発見)は維持した上で、利用の姿を次のように再定義する:

**「オーナーが『これ、友達に使わせたい。月 $30 まで』と言うだけで、オペレーター(エージェント)が sekimori を検出・設置・検証・運用し、オーナーには『何が守られているか』の平易な要約だけが届く」**

分解すると:

1. **エージェントが読者**: 一次ドキュメントは、エージェントが誤読なく実行できる形(決定的なコマンド、期待出力、終了コード)で存在する
2. **エージェントが操作者**: すべての操作(config 生成・起動・診断・トークン発行)が TTY なし・対話なし・推測なしで完結する
3. **エージェントが検証者**: 「守りが効いているか」を、デプロイの前後で機械可読に検証できる
4. **人間は承認者**: 人間に残る操作は「予算額を決める」「API キーと課金アカウントを差し出す」「要約を読む」の 3 つだけ。そこで見せるものは専門知識を仮定しない

## 3. 問題点と真因

### 観測される問題点(v0.2 完了時点)

| # | 問題 | 乖離する理想条件 |
|---|---|---|
| P9 | ドキュメントはすべて「人間の開発者が読む」形。エージェント向けの運用手順書(AGENTS.md 等)がなく、エージェントは README の散文から手順を再構成する必要がある | 条件 1 |
| P10 | `sekimori init` は対話が一級市民。非対話(`--yes`)は**全項目デフォルト固定**しか書けず、port・モデル・予算などを指定するフラグが存在しない。エージェントは結局 JSON を手組みする(可能だが、検証済みの経路を通らない) | 条件 2 |
| P11 | 「守りが効いているか」の検証手段が、人間向けの起動サマリ(散文ログ)と demo.sh(開発リポジトリ専用)しかない。デプロイ済み環境で config+env を機械可読に自己診断する手段がない | 条件 3 |
| P12 | v0.3 の計画(ドッグフーディング・デプロイガイド)が「人間がデプロイを実行する」前提で書かれている。新前提ではオーナーはデプロイを知らないので、この計画は実行者不在で永久に止まる | 条件 2・4 |
| P13 | README・コンセプトの語りが「あなた(開発者)が守る」であり、「あなたのエージェントが設置し、あなたの財布が守られる」という主要ユースケースが言語化されていない | 条件 4・発見の誠実さ |

### 真因

- **R5: 「読者=人間の開発者」という暗黙の前提が、v0.2 の全成果物に焼き付いている。** P9・P10・P13 は独立の欠陥ではなく、第 1 ラウンドが「配る相手」を人間に固定したまま最適化した帰結。英語化も README 整地も正しかったが、宛先が一世代前だった。
- **R6: 「検証」がリポジトリ内(テスト・CI・demo)に閉じており、稼働環境に持ち出せない。** P11。CI は「コードが正しい」を証明するが、エージェントが本番で必要なのは「**この設置**が正しい」の証明。両者は別物。
- **R7: 人間ゲートの定義が「人間が作業する」になっている。** P12。新前提での人間の希少資源は作業時間ではなく**権限(credential)と承認**。ゲートの定義を「人間がやる」から「人間が鍵を差し、エージェントがやる」に変えない限り、v0.3 は進まない。

## 4. 解決方針

| 真因 | 方針 | 具体物 |
|---|---|---|
| R5 | **エージェントを第一級の読者・操作者に昇格させる** | (a) `AGENTS.md`: 機械実行可能な運用手順書(インストール→config→起動→検証→トークン発行→監視。各ステップに期待出力と終了コード)。npm パッケージにも同梱。(b) `sekimori init` の完全非対話化: 全設定に対応するフラグ(`--port` `--model` `--monthly-usd` 等)+ `--help`。(c) README/コンセプトの語りに「エージェントが設置する」ユースケースを明記 |
| R6 | **設置の自己診断を道具に内蔵する** | `sekimori doctor <config>`: config の検証・必須環境変数の存在・ストア書き込み可否を非対話で診断し、人間向けの平易なサマリと `--json`(機械可読)の両方を出力。終了コードで合否を返す。エージェントはデプロイ前後にこれを実行し、オーナーへの報告には人間向けサマリを流用する |
| R7 | **人間ゲートを「credential ゲート」に再定義する** | v0.3 実施項目(実デプロイ検証・デプロイガイド)を「エージェントが実行し、人間は課金アカウントと API キーを提供して結果を承認する」形に書き直す。「実測なしのデプロイ手順は書かない」原則は維持(実行者がエージェントに変わるだけ) |

やらないこと(スコープ防衛): MCP サーバー化・エージェント用 SDK・自動デプロイ機能の内蔵は今回のスコープ外。sekimori は CLI と HTTP API という**既にエージェントが最も得意とする面**だけを磨く。プロトコルを増やすのは、実際のエージェント運用で不足が実証されてからでよい。

## 5. 目標設定(改訂)

**ゴール G′: 「オーナーは予算と鍵を差し出すだけ。オペレーター(エージェント)が sekimori の設置・検証・運用を人間の介在なしに完遂し、オーナーには平易な保護サマリが届く」**

- **v0.3 "agent-ready"(エージェント作業で完結・今回実施)**: AGENTS.md、init 完全非対話化、`sekimori doctor`、README/ROADMAP の位置づけ更新
- **v0.4 "public release"(credential ゲート)**: 実環境検証(エージェント実行・人間は credential 提供)→ 実測に基づくデプロイガイド → 商標確認 → npm publish

実行体制は不変: 設計・レビュー・監督 = Fable 5、実装 = Sonnet 5、承認と credential = 人間。
