# オーナー・オンボーディング・レビュー — 「credential を差し出す」が知識を要求していた

実施: 2026-07(設計・レビュー: Fable 5)。対象は v0.3 "agent-ready" 完了時点の sekimori 全体。

## 0. きっかけ: オーナー本人からの 2 つの質問

v0.3 完了報告は「残る人間の作業はアカウントを 2 つ差し出して承認するだけ」と締めた。それに対しオーナー(yktsnd、アプリ開発初心者)から返ってきたのは、次の 2 つの質問だった:

1. 「**ホスティングアカウントとは何でしょう?**」
2. 「**Anthropic キーとは何でしょう? AWS の初回登録でもらえる AWS Bedrock Free Credit が使えると嬉しいんですが……**」

この 2 問は苦情ではなく、**プロジェクトの前提の欠陥を突いた実測データ**である。本レビューは「なぜこの質問が生じたのか」を真因まで掘る。

## 1. なぜこの質問が生じたのか

### 観測

第 2 ラウンド([06-agent-operator-review.md](06-agent-operator-review.md))は登場人物を 3 役(オーナー/オペレーター=エージェント/エンドユーザー)に分け、「オーナーはデプロイも課金も知らない」と正しく定義した。しかし v0.3 の成果物を読者別に棚卸しすると:

| 読者 | 用意された文書 |
|---|---|
| 開発者(人間) | README、docs/configuration・api・design |
| オペレーター(エージェント) | AGENTS.md |
| コントリビューター | CONTRIBUTING、SECURITY、テンプレート |
| **オーナー** | **なし** |

オーナーは「第一級の役割」と宣言されながら、**唯一、自分宛の文書を持たない登場人物**だった。オーナーに触れる文言は AGENTS.md の「報告テンプレート」(=設置**後**にエージェントがオーナーへ送る文面)だけで、設置**前**にオーナーが果たすべき唯一の役割 — credential の提供 — を助けるものは何もない。

### 質問 1 が示すもの(ホスティングアカウント)

v0.4 の credential ゲートは「**[credential gate: hosting account + API key]**」という**名詞の列挙**で書かれている。名詞の列挙は「読者がその名詞を解決できる」ことを暗黙に仮定する。初心者のオーナーには「ホスティング」という概念自体(自分の PC ではない、インターネットから常時見える場所でプログラムを動かし続けるサービス)が未知であり、名詞は解決できない。**ゲートを開ける鍵の入手方法が、ゲートのこちら側に書かれていない。**

### 質問 2 が示すもの(Anthropic キー / Bedrock クレジット)

前半(「Anthropic キーとは?」)は質問 1 と同型の知識ギャップ。後半(「Bedrock Free Credit が使えると嬉しい」)はより深い:

- プロジェクトは上流を**プロトコル**(Anthropic Messages API)で定義してきた
- しかしオーナーは上流を**経済**(自分がすでに持っているクレジット・契約)で選ぶ
- Anthropic 直結のみの現状は、AWS クレジットを持つオーナーに「別途 Anthropic でプリペイド購入せよ」と要求することになる。これは想定ユーザー(お金の仕組みに不安のある個人)にとって最大級の導入障壁である

そして技術的事実として、この障壁を守る理由は薄れている。2025 年 7 月以降、Amazon Bedrock は **API キー(Bearer トークン)認証**を提供しており(`AWS_BEARER_TOKEN_BEDROCK`)、SigV4 署名なしで `bedrock-runtime.<region>.amazonaws.com/model/<modelId>/invoke` を呼べる。リクエスト本文は Anthropic Messages API とほぼ同形(`model`/`stream` を除き、`anthropic_version: "bedrock-2023-05-31"` を付与)、非ストリーム応答は Anthropic 形式そのもの。つまり **sekimori の設計(ヘッダ認証・依存ゼロ・素通し)のまま、非ストリーム経路は小さな変換で Bedrock に対応できる**。唯一の本質的差異はストリーミング(SSE ではなく AWS イベントストリームのバイナリフレーミング)で、これは変換実装が重く、別段の設計を要する。

## 2. 真因

- **R8: オーナーという役割を定義しながら、オーナー宛の成果物を作らなかった。** 第 2 ラウンドの盲点。読者別ドキュメント体系(R3 の解決)を「技術的読者」までで止め、非技術の第 4 の読者を落とした。
- **R9: credential ゲートが「名詞の列挙」で仕様化されており、「入手手順」を伴っていない。** ゲートの定義が誤っていた: ゲートとは「credential 名」ではなく「credential + その入手経路(それが何か・なぜ要るか・どう取るか・いくらかかるか・どう安全に渡すか)」の組である。
- **R10: 上流対応をプロトコルの都合で定義し、オーナーの経済(手持ちのクレジット)で定義していなかった。** 「Anthropic Messages API 互換のみ」は実装契約としては正しかったが、オーナーの語彙では「あなたの AWS クレジットは使えません」を意味する。Bedrock の API キー認証の登場により、この制約の技術的根拠は非ストリーム経路について消滅している。

## 3. 解決方針

| 真因 | 方針 | 具体物 |
|---|---|---|
| R8 | **第 4 の読者に文書を渡す** | `docs/owner-guide.md` + `docs/owner-guide.ja.md`(オーナーの一次言語が日本語であるため日英併設。非技術者向けに: API キーとは何か・Anthropic での取得手順と費用・ホスティングとは何か・代表的サービスと費用感・エージェントへの安全な渡し方・自分に残る判断は何か・FAQ)。README/AGENTS.md からリンクし、エージェントの報告テンプレートにも案内を組み込む |
| R9 | **ゲートの再定義** | v0.5(旧 v0.4)の credential ゲート issue を「オーナーガイドの該当節を読む→取得する→渡す」の誘導手順つきに書き直す |
| R10 | **Bedrock 上流対応(非ストリーム)** | `upstream.type: "anthropic" \| "bedrock"`。Bedrock 時は Bearer 認証(`AWS_BEARER_TOKEN_BEDROCK` 推奨)・`/model/<id>/invoke`・本文変換。**ストリーミングは fail-closed で明確に拒否**(イベントストリーム→SSE 変換は別 issue の Later 項目に)。参照クライアント chat.html に `CONFIG.stream` を追加し両上流で動くようにする。`init --upstream-type` と doctor は追随 |

**スコープ防衛の確認**: これは「100+ プロバイダ対応」(非目標)への一歩ではない。sekimori は **Claude 専用**のままであり、Bedrock は同じ Claude への別の入口にすぎない。上流プロバイダは当初から公認の拡張点であり、今回はその拡張点が「実利用からの引力」(オーナー本人のユーザーストーリー)で初めて行使される、ロードマップの想定どおりの動きである。

## 4. 目標設定(改訂)

**ゴール G‴: 「オーナーは予備知識ゼロでよい。自分宛のガイド 1 本(母語)で、必要な credential が何か・どう取るか・いくらかかるかを 10 分単位で理解でき、手持ちのクレジット(Anthropic 直またはAWS Bedrock)をそのまま使える。それ以外のすべてはエージェントがやる」**

- **v0.4 "owner-ready"(エージェント作業で完結・今回実施)**: オーナーガイド(日英)、Bedrock 上流(非ストリーム)、ドキュメント追随、credential ゲートの再定義
- **v0.5 "public release"(credential ゲート・旧 v0.4 を改番)**: 実環境検証 → デプロイガイド → 商標 → npm publish。オーナーガイドにより、ゲートを開ける知識面の前提が初めて満たされる
- **Later 追加**: Bedrock ストリーミング(イベントストリーム→SSE 変換。実装が重く、非ストリームで実需が確認されてから)
