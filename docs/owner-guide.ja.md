# オーナーガイド — 予備知識ゼロで読める、あなたの役割のすべて

このページは、**アプリを作る人ではなく、アプリの持ち主(オーナー)のための**説明書です。プログラミングやサーバーの知識は一切仮定しません。English version: [owner-guide.md](owner-guide.md).

あなたのアプリは、あなた自身か、あなたが指示したコーディングエージェント(Claude Code など)が作り、公開作業もエージェントが代行できます。その過程で**あなたにしかできないことは 3 つだけ**です:

1. **AI の利用料を払う口座(API キー)を用意する**
2. **アプリを動かす場所(ホスティング)のアカウントを用意する**
3. **決める**: 月にいくらまで使ってよいか・誰を招待するか

それぞれ「そもそも何なのか」から説明します。

---

## 1. API キーとは何か(「Anthropic キー」の正体)

AI(Claude)は、あなたのパソコンの中ではなく、提供元(Anthropic 社や Amazon)のコンピューターで動いています。アプリが AI に質問を送るたびに、提供元は「これは誰からの依頼で、誰に請求すればいいのか」を知る必要があります。その身元証明に使われる長い文字列(例: `sk-ant-api03-...`)が **API キー**です。

つまり API キーとは、**「あなたの支払いにつながる、AI 利用のための鍵」**です。だからこそ:

- キーを持つ人は誰でも**あなたのお金で** AI を使えます。**絶対に SNS やチャットに貼らない**でください
- sekimori の存在理由がまさにこれです: キーをサーバーの中だけに置き、アプリの利用者には**招待トークン**(上限つきの通行手形)だけを配ります。プロバイダーキーはエンドユーザーには渡しません

### 入手方法は 2 通りあります(どちらか 1 つでよい)

**方法 A: Anthropic から直接**

1. [console.anthropic.com](https://console.anthropic.com) でアカウントを作る(メールアドレスがあればよい)
2. アカウントに必要な支払い方法・credit を登録する。公開前に Anthropic の現在の公式 billing 案内を確認し、利用できる支出制御と auto-reload の動作を意図的に選ぶ
3. 「API Keys」のページで「Create Key」を押し、表示された `sk-ant-...` を提供元の現在の secret 保管案内に従って保存する（完全な値は作成時しか表示されない場合がある）
4. アカウントで現在利用できる提供元側の支出・billing 制御を、sekimori と独立した層として設定する

費用は、選ぶモデル・その時点の価格・設定した価格表・利用量で変わる。公開前に提供元の現在の価格と請求設定を確認すること。sekimori は対応するテキスト専用リクエストに対する追加の保護境界であり、提供元の請求設定の確認や価格表を正しく保つことの代わりにはならない。

**方法 B: Amazon Bedrock 経由(AWS のクレジットを持っている人向け)**

AWS クレジットが Bedrock に使える条件を満たしていれば、Amazon の AI サービスである Bedrock の利用に充てられる場合がある。Bedrock 上でも Claude が動いており、sekimori は小規模プロトタイプ向けに Bedrock にも対応している。クレジットの対象・期限・Marketplace の条件は、自分の AWS アカウントで確認すること。

1. [AWS アカウント](https://aws.amazon.com/jp/)を作る(既にあればそれでよい)
2. AWS の管理画面で使う Anthropic モデルを選び、その時点のモデルアクセス条件に従う。Anthropic モデルでは初回の用途フォーム、Marketplace 権限、リージョンやアカウント設定が必要になる場合がある
3. この連携用の Bedrock API キーを発行し、ホスティング側の Secrets / 環境変数にだけ保存する
4. キーの有効期限を記録し、期限前に更新する。sekimori は静的な環境変数を読むだけで、AWS の認証情報を自動更新しない

**認証に関する重要な制約**: AWS は長期 Bedrock API キーを探索用途のための簡便な方法と位置づけ、より強いセキュリティが必要なアプリケーションには短期認証情報を推奨している。sekimori は現在 static Bearer キーを使うため、Bedrock モードはプロトタイプ向けであり、長期運用の本番認証設計ではない。より強い保証が必要なら Anthropic 直接接続か、短期認証情報を更新できるデプロイ/認証設計を使う。

**ストリーミングの制約**: Bedrock 経由では、現在の sekimori は「文字がぽつぽつと流れるように表示される」方式(ストリーミング)が使えず、**返答が一括で表示される方式のみ**です。チャットとしては問題なく使えますが、体感は少し変わります(対応予定はロードマップ参照)。

> Bedrock はアカウント・アクセス・請求・リージョン・キー更新の条件が用途に合うと確認できた場合に選ぶ。Anthropic 直接接続ではストリーミングを使え、Bedrock では現在使えない。実際に確認した要件で選ぶ。

## 2. ホスティング(アカウント)とは何か

あなたのパソコンでアプリを動かしても、パソコンを閉じれば止まり、他人はアクセスできない。**インターネット上に常時つながった、借り物のコンピューターでプログラムを動かし続けるサービス**をホスティングと呼ぶ。「ホスティングアカウントを用意する」とは、そうしたサービスに**メールアドレスで登録し、必要なら支払い方法を登録する**こと。現在の料金、secret 保管、HTTPS、永続 volume、log、1 replica の設定を確認してから選ぶ。sekimori には、まだ実配置から検証した hosting recipe はない。

- どのサービスを選ぶか・その後の設定は、候補・アカウント・発生し得る料金をあなたが承認した後で operator に任せられる。アカウント作成前に、現在の公式料金と検証根拠を示すよう依頼する
- ここで登録するカードは「アプリを置く場所代」、方法 A/B のキーは「AI の利用料」で、**別々の支払い**です

## 3. あなたが決めること(エージェントに聞かれたら答えること)

| 質問 | 意味 | 決め方 |
|---|---|---|
| 月の上限はいくら? | 全員合計の設定上の月次利用枠。対応 request は保守的な予約がローカル計算上限を超える前に停止する。提供元の請求制御も有効にする | 自分が許容できる最大額を自分で決める。プロジェクトの推奨初期値はなく、agent に決めさせない |
| 1 人あたり 1 日いくら? | 1 招待 token が UTC の 1 日に使える設定枠 | 招待人数と月額から配分し、自分で承認する |
| 誰を招待する? | 招待コード(トークン)を渡す相手。1 人ずつ即時に取り消せます | — |

## 4. キーを安全にエージェントへ渡すには

API キーは「秘密の値」です。渡し方だけ注意してください:

- **よい渡し方**: ホスティングサービスの「環境変数」「Secrets」と呼ばれる設定欄に自分で貼り付ける(エージェントが場所を案内します)。または、エージェントが用意した安全な設定手順に従う
- **避ける渡し方**: 公開される場所(SNS、共有されるチャットログ、アプリのコードの中)に貼ること
- 万一漏れた場合は、発行元の管理画面(Anthropic Console / AWS)でそのキーを**無効化**し、環境変数の値も更新する。sekimori の上限は追加の保護になるが、漏えいは緊急事態として扱い、提供元側の請求・アクセス制御も確認する

## 5. 設置が終わったら、何を確認すればいいか

エージェントは設置後に、次のような**保護の要約**をあなたに報告します(sekimori の自己診断機能 `doctor` が自動生成する内容に基づきます):

> - 使えるモデル: <確認済みの許可モデル>
> - 月の上限: 全員合計 $<承認した額> / 1 人 1 日 $<承認した額>
> - 1 人あたり rolling minute で <設定値> 回まで
> - 会話の中身は記録されない
> - provider key と admin key はサーバー側に設定されている

最低限、**自分が決めた数字と完全に一致するか**、モデルと価格が operator の再確認したものかを確認する。この要約は技術監査の代わりではない。[AGENTS.md](../AGENTS.md) の `doctor` 成功結果と live blocked/allowed check も提示させ、提供元の請求制御も確認する。数字が違う場合は招待 token を発行せず、承認した正確な値を伝えて検証をやり直す。

## よくある質問

**Q. AI の請求が青天井にならないか心配です。**
A. 1 つではなく複数の層で守る。(1) 対応するテキスト専用リクエストに対して、現在の価格と正しい上限で sekimori を設定する。(2) 提供元の請求設定を確認する。(3) Anthropic では、追加クレジットを買いたくなければ auto-reload をオフにする。AWS では予算アラートとアカウント側の支出制御を設定する。どれか 1 つだけで他を置き換えることはできない。

**Q. AWS クレジットは Bedrock に使えますか?**
A. 使える場合があるため、クレジットとアカウントが対象か確認する。sekimori はプロトタイプ用に Bedrock API キー認証に対応しているが、static キー連携は短期認証情報を自動更新せず、返答の「流れる表示」(ストリーミング)にも非対応で一括表示になる。

**Q. 招待した友人が使いすぎたら?**
A. 対応 request subset と正しく宣言したモデル価格の範囲では、保守的な予約がその token の日次計算上限を超える前に停止し、翌日(UTC)に再開する。予約が設定した全体の月次計算上限を超える場合は全 token が翌月まで停止する。提供元側の請求制御も独立した層として有効にしておく。

**Q. アプリを止めたくなったら?**
A. エージェントに「止めて」と言えば、ホスティング側でアプリを停止できます。API キーを無効化すれば AI 利用も完全に止まります。

**Q. このガイドの英語版・技術的な詳細は?**
A. 英語版は [owner-guide.md](owner-guide.md)。技術的な仕組みは [README](../README.md)(開発者向け)と [AGENTS.md](../AGENTS.md)(エージェント向け)にあります。あなたが読む必要はありません。
